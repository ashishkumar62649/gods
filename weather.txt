| --------------------------------- | -------------------------------------------------------------------------------- |
| Do not fetch all sources together | Each source needs its own schedule.                                              |
| Use priority queues               | Tier 1 gets high-priority workers; Tier 5 gets cold/background workers.          |
| Use adaptive polling              | Increase fetch rate only during active danger.                                   |
| Use deduplication                 | If event ID exists, update it; otherwise insert it.                              |
| Use raw storage first             | Save original API/file response before normalization.                            |
| Use source-specific limits        | Respect API key, token, rate limit, and terms.                                   |
| Use model-cycle fetching          | For GFS/CAMS/GloFAS, fetch after new model files are released.                   |
| Split metadata from live values   | Station lists, volcano lists, sensors change slowly; measurements change faster. |
| Cache tile sources                | NASA GIBS/GOES visual tiles should be cached/on-demand, not blindly downloaded.  |

3. Always partition by time (year/month/day/hour)
4. Always store original format (no conversion here)

6. Never mix multiple sources in one folder
7. Large data (GRIB/NetCDF) stays exactly as downloaded
8. Small data (JSON/CSV) stored directly or gzip if needed
9. Satellite tiles → store reference or cache, not full dump
10. Errors/debugging → isolated per source folder

Different sources return:
- Tiles (PNG/JPG via WMTS/WMS)
- Hybrid (API + WMS)
- Raster (GeoTIFF)
- Structured (JSON/CSV/GeoJSON)
- Binary grids (GRIB2/NetCDF)

GOAL:
Unify ALL into ONE system → "queryable + visualizable + normalized"

  era5

  gpm

────────────────────────────────────────────
STEP 2: STORAGE STRATEGY (UNIFIED)
────────────────────────────────────────────

TYPE_A (tiles)
→ DO NOT STORE
→ store only:
   - layer_name
   - url_template
→ render directly in frontend (Cesium)

TYPE_B (hybrid)
→ store BOTH:
   1. raw API data → /data/raw/
   2. WMS URL → config
→ use:
   - API → intelligence
   - WMS → visualization

TYPE_C (GeoTIFF)
→ store raw GeoTIFF
→ convert → COG OR Parquet grid
→ extract:
   - value
   - lat/lon
   - h3 index

TYPE_D (events)
→ raw JSON → /data/raw/
→ normalized → PostgreSQL (intel_events)

TYPE_E (observations)
→ raw JSON → /data/raw/
→ normalized → PostgreSQL (intel_observations)

TYPE_F (grid models)
→ raw GRIB/NetCDF → /data/raw/
→ convert → Parquet (DuckDB readable)
→ DO NOT store in PostgreSQL

────────────────────────────────────────────
STEP 3: NORMALIZATION PIPELINE PER TYPE
────────────────────────────────────────────

TYPE_A:
  skip normalization
  just store config

TYPE_B:
  normalize API data (like TYPE_D/E)
  ignore WMS for backend

TYPE_C:
  GeoTIFF → rasterio/gdal
  → extract pixels
  → map to H3
  → save as Parquet

TYPE_D:
  JSON → map fields
  → add H3
  → insert into events table

TYPE_E:
  JSON → standardize units
  → add H3
  → insert into observations table

TYPE_F:
  GRIB/NetCDF → xarray
  → select variables
  → convert to Parquet
  → index by H3 or grid cell

────────────────────────────────────────────
STEP 4: QUERY UNIFICATION
────────────────────────────────────────────

User query:
→ location (lat/lon)

Backend:
→ convert to H3

Then:

PostgreSQL:
→ events
→ observations
→ assets

DuckDB:
→ weather grids (Parquet)

Frontend:
→ tiles (GIBS/WMS)

Combine:
→ single JSON response

────────────────────────────────────────────
STEP 5: VISUALIZATION STRATEGY
────────────────────────────────────────────

Events:
→ points (Cesium entities)

Observations:
→ markers / heatmap

Grids:
→ raster layer OR sampled values

Tiles:
→ direct overlay (WMTS/WMS)

────────────────────────────────────────────
STEP 6: KEY RULES
────────────────────────────────────────────

1. NEVER convert everything to one format
2. KEEP each type in optimal format
3. UNIFY only at query time
4. USE H3 as join key
5. USE DuckDB for large grids
6. USE Postgres for relationships
7. USE frontend for tile rendering

────────────────────────────────────────────
FINAL SYSTEM MODEL
────────────────────────────────────────────

Tiles (GIBS) → Frontend
Events → PostgreSQL
Observations → PostgreSQL
Grids → Parquet + DuckDB
Raster → GeoTIFF → Parquet
Raw → MinIO

ALL JOINED BY:
→ H3 index
→ time
